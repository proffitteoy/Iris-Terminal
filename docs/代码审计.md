# 代码审计

更新日期：2026-04-29

## 审计范围

- `app/api/local/*`
- `app/api/memory/retrieve/route.ts`
- `app/api/storage/[...path]/route.ts`
- `db/*`
- 本地 Bootstrap 与存储链路

## 结论

本仓库当前主风险不在“聊天主链路不可用”，而在以下三类问题：

1. `db/*` 中存在大量占位实现，返回 `null/[]/true`，会把未落地能力伪装成成功。
2. 文档缺失，导致“类型面、UI 面、Prisma 实现面”长期不一致。
3. 部分边界处理不够严格，如 JSON null 过滤类型不一致、MIME 类型声明缺失。

本轮已按“文档补齐 + fail-closed + 类型修复”处理。

## 关键发现与处理

### 1. 未落地实体被隐式信任

受影响模块：

- `db/assistants.ts`
- `db/collections.ts`
- `db/models.ts`
- `db/presets.ts`
- `db/prompts.ts`
- `db/tools.ts`
- 以及若干关系表包装模块

问题：

- 调用方把这些函数当成真实数据层
- 但实现只返回 `null`、空数组或 `true`
- 这会掩盖缺失的本地 API / Prisma 支撑

处理：

- 保留工作区初始化需要的空集合读取
- 将写操作与关键读操作改成显式失败
- 用统一错误类型收口

修复的信任问题：

- 修复“函数存在即能力已实现”的错误假设
- 修复“返回 true 即写入成功”的错误假设

### 2. 文档与实现严重脱节

问题：

- 根目录缺少 README
- `docs/` 中只有通用说明和提示词，缺少仓库真实架构、API、能力边界

处理：

- 新增 README 与仓库级文档
- 明确哪些能力已支持、哪些仍关闭

修复的信任问题：

- 修复“文档默认等于实现”的错误假设

### 3. Prisma JSON null 过滤不严

受影响模块：

- `app/api/memory/retrieve/route.ts`

问题：

- 对 JSON 字段使用 `not: null`
- 在当前 Prisma 类型下不安全，且 `tsc` 已报错

处理：

- 改为 Prisma 提供的显式 null 过滤常量

修复的信任问题：

- 修复“TypeScript 能过 / null 写法通用”的错误假设

### 4. MIME 类型声明缺失

受影响模块：

- `app/api/storage/[...path]/route.ts`

问题：

- `mime-types` 缺失本地声明，导致类型检查失败

处理：

- 补充最小声明文件

修复的信任问题：

- 修复“运行能用就等于类型系统认可”的错误假设

## 剩余风险

- `/api/chat` 新入口当前以“检索编排 + 结构化答案输出”为主，尚未统一接入 provider 流式生成
- Knowledge 向量检索链路已预留 `pgvector` 字段，但在线 embedding 生成与召回策略仍需后续接入
- 新增知识层迁移已落库到 `prisma/migrations/*`，但需在目标环境执行迁移后才会生效

## 后续建议

1. 若要统一 `/api/chat`，建议在当前编排入口下追加 provider adapter 与流式输出协议
2. 给 `/api/knowledge/sync` 与 `/api/knowledge/search` 增加最小集成测试（含路径校验、删除标记、引用可追踪）
3. 若要恢复 assistants / collections 等能力，先补 Prisma 模型与 `/api/local/*`
