## 1. 阶段目标

二阶段目标不是“做一个新知识库页面”，而是把 **Chat-first 的知识增强闭环** 做到可用、可验、可追踪：

1. 用户在 Chat 中提问时，系统可基于本地知识源执行检索增强。
2. 回答必须附带可追踪来源（`sources/citations`），支持前端渐进披露。
3. 查不到时失败闭合，不允许“无依据编造”。

## 2. 范围与边界

### 2.1 二阶段范围（In Scope）

- 数据与索引基线：
  - `Vault / Note / NoteChunk / NoteLink / RetrievalEvent` 模型与迁移
  - Vault 扫描、解析、chunk、link 与索引同步
- 检索与编排：
  - `POST /api/knowledge/search`（analyzer + hybrid + graph expand + rerank）
  - `POST /api/chat`（Analyzer -> Retrieval -> Context -> Citation）
- 前端披露：
  - 回答来源胶囊与来源抽屉（SourceChips / SourceDrawer）
  - 来源原文片段预览与基础调试信息
- 写回闭环：
  - `POST /api/knowledge/actions` 写回 Obsidian `90_AI_Drafts`

### 2.2 二阶段非范围（Out of Scope）

- `/api/chat` 全 provider 流式统一入口改造（保留现有 provider 路由）
- 完整自动检索决策策略产品化（Phase 3）
- 大规模图谱可视化与复杂图分析（Phase 5）
- 未落地实体（assistants/collections/presets/prompts/tools/models）恢复

## 3. 工作分解结构（WBS）

### WBS-1：数据与同步基线

目标：保证“知识可入库、可增量更新、可追踪来源”。

关键任务：

1. 校准 Prisma 模型与迁移脚本。
2. 完成 Vault 增量同步（扫描、hash 检测、更新、删除标记）。
3. 统一 note/chunk/link 的写入一致性约束。

阶段产物：

- 可执行迁移脚本
- `POST /api/knowledge/sync` 可重复运行
- notes/chunks/links 数据一致

### WBS-2：检索与路由编排

目标：保证“检索命中可解释，失败路径可控”。

关键任务：

1. 落地 Query Analyzer 与 Retrieval Router。
2. 完成全文检索 + 图扩展 + rerank 的编排与阈值控制。
3. 落库 RetrievalEvent（请求、策略、命中、耗时、降级路径）。

阶段产物：

- `POST /api/knowledge/search` 稳定返回候选片段与 citations
- 检索过程可在日志或调试面板复盘

### WBS-3：Chat Grounded 回答链路

目标：保证“回答和证据绑定输出”。

关键任务：

1. 将检索结果通过 Context Composer 注入 `/api/chat`。
2. 固化 `answer + sources + citations` 的输出契约。
3. 在无命中或低置信度场景下执行 fail-closed 响应策略。

阶段产物：

- `POST /api/chat` 可返回结构化证据元数据
- 低命中场景不编造来源

### WBS-4：前端渐进式披露

目标：默认不打扰，按需展开证据细节。

关键任务：

1. 回答下方展示来源胶囊（命中数量、主要笔记）。
2. SourceDrawer 展示原文片段、笔记路径、关联信息。
3. 增加最小调试面板字段（检索模式、命中数、耗时）。

阶段产物：

- 用户可从回答跳转到证据片段
- 调试信息可用于研发排障

### WBS-5：质量与发布门禁

目标：保证“可回归、可维护、文档一致”。

关键任务：

1. 补齐接口与核心链路测试（至少覆盖 sync/search/chat/actions 主路径）。
2. 运行 `npm run lint`、`npm run type-check`、`npm run db-generate`。
3. 同步更新文档（API、能力边界、架构说明、README 导航）。

阶段产物：

- 阶段验收记录
- 文档与实现一致的发布基线

## 4. 执行顺序与依赖

推荐顺序（严格按依赖推进）：

1. WBS-1 数据与同步基线
2. WBS-2 检索与路由编排
3. WBS-3 Chat Grounded 回答链路
4. WBS-4 前端渐进式披露
5. WBS-5 质量与发布门禁

说明：后置工作包不能绕过前置数据约束和接口契约，否则会出现“UI 可见但后端不可验证”的伪完成状态。

## 5. 里程碑与验收标准

### M1：索引基线可用

- 可注册 Vault 并成功同步 Markdown 到 notes/chunks/links
- 同一 Vault 重复同步结果稳定，不出现重复脏写

### M2：检索编排可解释

- `/api/knowledge/search` 返回可追踪候选与 citations
- 检索事件可复盘策略与耗时

### M3：Grounded Chat 可闭环

- `/api/chat` 返回结构化 `answer + sources + citations`
- 前端可查看来源片段，且无命中时不编造

### M4：发布门禁通过

- lint/type-check/db-generate 通过
- API说明、能力边界、README 导航完成同步

## 6. 风险与应对

1. 检索质量波动：
   - 应对：保留词法回退与阈值控制，记录 RetrievalEvent 便于调参。
2. 同步性能与一致性：
   - 应对：增量 hash + 批量写入，建立删除标记与重跑机制。
3. 文档与实现漂移：
   - 应对：将文档更新纳入 M4 强制门禁，不满足即不验收。
4. 未落地能力被误开放：
   - 应对：沿用 fail-closed 规则，写操作显式报错。
