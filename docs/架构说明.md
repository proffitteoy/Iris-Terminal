# 架构说明

更新日期：2026-04-29

## 1. 总体定位

`鸢尾花终端` 当前是一个本地优先的单体 Web 应用：

- 前端：Next.js 14 App Router
- 服务端：Next.js Route Handlers
- 数据库：Prisma + PostgreSQL
- 文件存储：本地文件系统 `storage/`
- 模型接入：OpenAI / Azure OpenAI / Anthropic / Gemini / DeepSeek 等

它不是完整复刻上游通用 Chat UI 的所有实体能力，而是保留本地部署场景里已经打通的主链路，并对未落地能力做显式边界控制。

## 2. 分层

### UI 层

- `app/[locale]/*`：页面布局、路由入口
- `components/*`：聊天界面、侧边栏、设置、工具组件
- `context/context.tsx`：全局 UI 状态容器

### 数据访问层

- `db/*`：前端对数据的访问入口
- 约束：
  - 已落地能力：调用真实 `/api/local/*` 接口
  - 未落地能力：显式抛出 `UnsupportedLocalFeatureError`

### 服务逻辑层

- `lib/local-bootstrap.ts`：本地用户 / Profile / 默认工作区引导
- `lib/retrieval/processing/*`：文件解析与切片
- `lib/server/*`：服务端聊天与配置辅助逻辑
- `lib/summaries/*`：总结格式、关键词提取

### 持久化层

- `prisma/schema.prisma`：当前本地模式实际落地的数据库模型
- `storage/`：文件二进制内容与图片资源

## 3. 当前真实数据流

### 3.1 启动与本地引导

1. 页面进入工作区布局
2. `ensureLocalBootstrap()` 自动创建或更新：
   - 固定本地用户
   - 本地 Profile
   - 默认工作区
3. UI 再通过 `db/*` 加载工作区、聊天、文件、总结等数据

### 3.2 聊天链路

1. UI 发送聊天请求到 `app/api/chat/*`
2. 消息写入 `messages`
3. 对话元数据写入 `chats`
4. 若触发总结，再由总结相关接口写入 `chat_summaries`

### 3.3 文件链路

1. 上传文件到 `app/api/local/files`
2. 文件原文写入 `storage/files/*`
3. 元数据写入 `files`
4. 切片写入 `file_items`
5. 向量按能力写入 `openai_embedding` 或 `local_embedding`

### 3.4 记忆检索链路

1. `app/api/memory/retrieve` 接收用户输入
2. 优先尝试 OpenAI / Azure Embedding
3. 若失败则降级到本地 Embedding
4. 若仍无结果则降级到词法匹配

## 4. 目录职责

### `app/`

- 页面与 API 路由
- `app/api/local/*` 是本地模式下最重要的真实数据接口

### `db/`

- 只能承担“请求编排与 I/O 适配”职责
- 不允许再放 `null/[]/true` 伪实现掩盖能力缺失

### `lib/`

- 放可复用逻辑
- 不把页面状态管理、数据库请求、存储 I/O 混进 React 组件

### `types/`

- 当前存在“类型面大于本地实现面”的历史遗留问题
- `types/database.ts` 覆盖的实体多于 `prisma/schema.prisma`
- 新增或启用实体能力时，要同时校验这两套定义是否一致

## 5. 当前架构红线

- 不新增只在前端伪装可用、但服务端未落地的实体能力
- 不允许 `db/*` 中的写操作静默成功
- 不允许在没有 Prisma 模型和本地 API 的情况下开启对应 UI 写入口
- 任何新能力都必须同步更新文档与能力边界清单
