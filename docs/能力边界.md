# 能力边界

更新日期：2026-04-29

本文件用于明确“哪些能力已经真实落地，哪些能力当前只是类型面残留或 UI 预留”。

| 能力 | 当前策略 | 原因 |
| --- | --- | --- |
| Assistants | 列表读空、写操作报错 | 本地 API 与 Prisma 模型未落地 |
| Collections | 列表读空、写操作报错 | 同上 |
| Presets | 列表读空、写操作报错 | 同上 |
| Prompts | 列表读空、写操作报错 | 同上 |
| Tools | 列表读空、写操作报错 | 同上 |
| Models | 列表读空、写操作报错 | 同上 |
| Folders 写操作 | 报错 | 当前主流程不依赖 folders 能力 |
| 文件多工作区写关联 | 报错 | 缺少本地 API 与持久化实现 |
| 总结客户端写接口 | 报错 | 仅保留服务端生成链路 |
| `/api/chat` 真实多模型流式推理 | 未并入新入口 | 仍由 `/api/chat/openai`、`/api/chat/deepseek` 等 provider 路由承载 |
| Knowledge 向量检索在线 embedding | 部分未启用 | 当前已落地全文 + 图扩展链路，向量 embedding 写入/召回待下一阶段接入 |

## 3. 为什么要这样收敛

过去仓库里存在大量 `return null`、`return []`、`return true` 的占位实现，问题是：

- 调用方会把它们当成真实成功结果
- TypeScript 会被迫接受大量 `null` 分支
- 文档和实际行为不一致
- 一旦 UI 打开入口，运行期行为不可预测

当前策略改成：

- 工作区加载必须依赖的“禁用能力列表”可显式返回空集合
- 任何写操作或关键读操作，必须失败闭合

## 4. 启用一个未落地能力前必须完成的事项

1. Prisma schema 补齐实体
2. `/api/local/*` 路由补齐
3. `db/*` 包装层改为真实调用
4. UI 入口再开放
5. 更新：
   - `docs/API说明.md`
   - `docs/架构说明.md`
